# 优化自检清单

> 体检与收尾时对照打钩。`[红线]` 表示必须先修；`[建议]` 表示按 ROI 排序优化；`[人工]` 表示脚本会提示但不计入自动分母。

## A. 优化前（锁定方向）
- [ ] `[建议]` 明确本次主攻维度：结构与上下文 / 触发与路由 / 任务契约 / 执行流程 / 输出稳定 / 运行稳定与故障恢复 / 工具化 / 评测回归 / 沉淀演进 / 安全边界 / 可维护性。
- [ ] `[建议]` 已运行 `health_check.py <skill目录>` 获取维度分。
- [ ] `[建议]` 触发类问题已运行 `health_check.py <skill目录> --skills-root <skills根目录>` 或 `audit_description.py <skills根目录>`。
- [ ] `[建议]` 改前已记录基线、版本号和可回滚点。

## B. 结构与上下文健康（12 分）
- [ ] `[红线]` 存在合法 `SKILL.md`。
- [ ] `[红线]` frontmatter 可解析，且包含非空 `name` / `description`。
- [ ] `[红线]` `SKILL.md` 中引用的强依赖文件真实存在。
- [ ] `[建议]` `SKILL.md` 只放决策骨架和资源索引，细节放入 `references/`。
- [ ] `[建议]` `SKILL.md` 主体建议小于 300 行，超过时拆分。

## C. 触发与路由质量（16 分）
- [ ] `[红线]` description 非空，能让 Codex 判断何时触发。
- [ ] `[建议]` description 建议 ≤1024 字符，核心触发词放在第一句前 80 字符。
- [ ] `[建议]` description 同时写“何时用”和“何时不用”。
- [ ] `[建议]` 存在 `agents/openai.yaml`，必要时可配置 `allow_implicit_invocation:false`。
- [ ] `[建议]` 与同根目录下其他 Skill 无明显触发词重叠；若重叠，已区分主次 Skill。

## D. 任务契约清晰度（9 分）
- [ ] `[建议]` 写清 INPUT：用户给路径、文件、文本、症状还是参数。
- [ ] `[建议]` 写清 OUTPUT：最终交付报告、文件、改动、清单还是命令结果。
- [ ] `[建议]` 缺参时知道何时继续、何时提问、何时降级。
- [ ] `[建议]` 明确适用范围和排除范围，避免把业务问题本身误当 Skill 优化。

## E. 执行流程可操作性（8 分）
- [ ] `[建议]` 有 SOP、路径 A/B/C 或步骤清单。
- [ ] `[建议]` 不同输入、症状和风险能路由到不同处理路径。
- [ ] `[人工]` 专家判断已显性化为 if-then、决策树或阈值。
- [ ] `[建议]` 工具失败、缺权限、缺文件时有降级方案。

## F. 输出稳定性（10 分）
- [ ] `[建议]` 有固定输出模板，字段占位符清晰。
- [ ] `[建议]` 至少有 1-2 个 Few-shot 或完整示例。
- [ ] `[建议]` 写了反例、禁止事项或“不要这样做”。
- [ ] `[建议]` 输出字段、顺序、格式和验收要求明确。
- [ ] `[红线]` 体检类输出必须是完整报告，禁止只给总分、评级或一句结论。

## G. 运行稳定性与故障恢复（10 分）
- [ ] `[建议]` 写清工具失败、缺目录、缺文件、缺权限、脚本报错时如何降级。
- [ ] `[建议]` 支持重复运行或复验，不会因为重复执行造成不可预期副作用。
- [ ] `[建议]` 关键外部依赖、前置权限、输入文件要求有明确说明。
- [ ] `[建议]` 部分检查未运行时会显式标注 MANUAL / SKIP，而不是静默忽略。
- [ ] `[建议]` 改动类任务说明备份、回滚、验证或失败恢复路径。

## H. 工具化与确定性（8 分）
- [ ] `[建议]` 数学、统计、字符数、触发重叠、格式转换、批量处理进入 `scripts/`。
- [ ] `[建议]` 脚本有 usage、参数说明或 argparse。
- [ ] `[建议]` 脚本处理编码、路径不存在、参数错误等常见异常。
- [ ] `[人工]` 纯指令 Skill 可不带脚本，但必须确认没有确定性计算需求。

## I. 评测与回归能力（10 分）
- [ ] `[建议]` 存在 Golden Set / eval / fixture 基准文件。
- [ ] `[建议]` 覆盖 5-10 个真实案例，至少包含健康对照和典型坏例。
- [ ] `[建议]` 改前改后跑同批案例，能量化提升或避免退化。
- [ ] `[建议]` 有最低分、命中率或必须命中的诊断项。

## J. 沉淀与演进（5 分）
- [ ] `[建议]` 存在 `SKILL.patch.md`。
- [ ] `[建议]` 有版本号、changelog 或变更记录。
- [ ] `[建议]` 失败案例、误报、漏报会回填到 patch 或 Golden Set。

## K. 安全与边界（8 分）
- [ ] `[建议]` 声明不适用场景和禁止事项。
- [ ] `[红线]` 涉及真实写操作时，必须写明确认、权限、备份或回滚边界。
- [ ] `[人工]` 涉及 token、账号、客户数据、上传下载时，必须写明敏感信息边界。
- [ ] `[建议]` 涉及删除、发送、发布、上传、覆盖等不可逆动作时，必须说明二次确认或拒绝边界。
- [ ] `[建议]` 遵守最小权限原则，不要求不必要的账号、token 或外部连接。
- [ ] `[建议]` 明确哪些场景必须拒绝执行或先询问用户。

## L. 可维护性（4 分）
- [ ] `[建议]` Skill name 使用小写 kebab-case。
- [ ] `[建议]` 元数据、依赖、触发开关放入 `agents/openai.yaml`。
- [ ] `[建议]` 关键资源使用明确相对路径，方便后续维护。
- [ ] `[建议]` 模板、方法论、脚本集中到标准目录。

---
**通过标准**：无红线项；总分 ≥90 为健康，80-89 为可用但建议优化，低于 80 应先修最高 ROI 问题。触发类任务优先看“触发与路由质量”，稳定性任务优先看“输出稳定性 + 评测与回归能力”。
