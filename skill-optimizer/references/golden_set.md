# Golden Set —— skill-optimizer 评测基准

> 方法 #12 落地。用「已知问题 Skill → 期望诊断结果」做回归测试，确保每次改 skill-optimizer 后诊断能力不退化。
> 用法：把下列 fixture 喂给 `health_check.py` / `audit_description.py`，对比实际输出与「期望命中」。

---

## 案例 1：健康 Skill（评分对照组）
**输入 fixture**：`skill-optimizer` 自身。

**期望诊断命中**：
- 总分 ≥90。
- 无红线项。
- 关键维度（结构与上下文、触发与路由、输出稳定、运行稳定性与故障恢复、评测与回归、安全与边界）均 ≥80。
- `--format json` 输出包含 `totalScore`、`dimensions`、`blockers`、`topFixes`、`optimizationPlan`。

## 案例 2：缺 SKILL.md
**输入 fixture**：一个空目录。

**期望诊断命中**：
- 红线项包含「缺 SKILL.md」。
- 等级为 `BLOCKED`。
- 结构与上下文健康为阻塞。
- Top ROI 修复建议第一项要求创建合法 `SKILL.md`。

## 案例 3：frontmatter 缺失或不合法
**输入 fixture**：
```markdown
# Broken Skill

没有 YAML frontmatter，只有正文。
```

**期望诊断命中**：
- 红线项包含「frontmatter 无法解析」。
- 红线项包含缺 `name` 或缺 `description`。
- 结构与上下文健康显著扣分。
- JSON 中对应检查项 status 为 `FAIL`。

## 案例 4：触发被截断的坏描述
**输入 fixture**：
```yaml
---
name: bad-trigger-skill
description: 这是一个非常专业的、融合多年经验的亚马逊广告分析工具，经过大量实践验证，能够处理各种复杂场景，帮助卖家深入理解数据并形成策略建议。它支持多种报告、多种指标、多种广告类型、多种推广阶段、多种诊断方式，并能在长期运营中持续沉淀方法论。当用户上传广告报表时清洗诊断。不适用于普通闲聊。
---
```

**期望诊断命中**：
- 触发与路由质量扣分。
- 命中「触发词前置」WARN。
- 建议使用倒金字塔重写，第一句直接写「清洗诊断亚马逊广告报表」。

## 案例 5：触发打架的两个 Skill
**输入 fixture**：两个 Skill 的 description 都含「listing 文案 标题 五点 描述 撰写 优化」。

**运行方式**：`health_check.py <其中一个skill目录> --skills-root <fixture根目录>`。

**期望诊断命中**：
- 触发与路由质量扣分。
- 命中「跨 Skill 触发词冲突审计」WARN。
- 建议次要 Skill 设置 `allow_implicit_invocation:false` 或重写 description。

## 案例 6：无输出模板的自由发挥 Skill
**输入 fixture**：一个只有泛泛流程说明、没有 `assets/`、没有「模板/字段/JSON/表格」约束的 Skill。

**期望诊断命中**：
- 输出稳定性扣分。
- 命中「有输出模板或 assets/」WARN。
- 命中「输出字段/格式约束明确」WARN。
- Top ROI 修复建议包含补固定模板。

## 案例 7：无 Golden Set 的 Skill
**输入 fixture**：结构、触发、模板都基本完整，但 `references/` 中没有 golden/eval/fixture/基准文件。

**期望诊断命中**：
- 评测与回归能力扣分。
- 命中「存在 Golden Set / 评测基准」WARN。
- 建议补 5-10 个真实案例 + 期望诊断。

## 案例 8：缺脚本的计算型 Skill
**输入 fixture**：一个声称「自动计算广告出价、预算、字符数、指标统计」但无 `scripts/` 的 Skill。

**期望诊断命中**：
- 工具化与确定性扣分。
- 命中「确定性计算已脚本化」WARN。
- 建议把数学、统计、格式转换和批量处理放入 `scripts/`。

## 案例 9：真实写操作缺权限边界
**输入 fixture**：一个说明会「直接修改、删除、上传、发送」但没有确认、权限、备份、回滚描述的 Skill。

**期望诊断命中**：
- 红线项包含「涉及真实写操作但缺少权限确认/回滚边界」。
- 安全与边界为阻塞。
- Top ROI 修复建议要求补确认、权限、备份或回滚边界。

## 案例 10：诊断类 Skill 只给总分
**输入 fixture**：一个用于「体检 / 诊断 / 审计」的 Skill，说明会输出评分，但没有完整报告结构，也没有声明禁止只给总分或评级。

**期望诊断命中**：
- 输出稳定性扣分。
- 命中「体检输出强制完整报告」FAIL。
- 红线项包含该检查项，Top ROI 修复建议要求补完整报告结构。
- 普通非诊断类 Skill 不应因缺少完整体检报告结构被误判红线。

## 案例 11：运行失败无降级路径
**输入 fixture**：一个依赖脚本、外部工具或权限的 Skill，但没有说明工具失败、缺目录、缺文件、缺权限、部分检查未运行时如何处理。

**期望诊断命中**：
- 运行稳定性与故障恢复扣分。
- 命中「工具失败/缺文件/缺权限有降级策略」WARN。
- 命中「部分失败或未运行项可见」WARN。
- 建议在报告中显式标注 MANUAL / SKIP 和失败原因。

## 案例 12：优化请求缺待确认执行计划
**输入 fixture**：用户要求「优化这个 Skill」，体检报告中存在 WARN/FAIL，但输出只列问题和建议，没有系统化待确认执行计划，也没有明确等待用户确认。

**期望诊断命中**：
- 最终交付必须包含系统化待确认执行计划。
- 计划至少包含阶段、优先级、改动对象、具体动作和验收方式。
- 计划输出后必须等待用户确认；确认前不得修改目标 Skill。
- 用户确认后，才按确认范围修改、复验并沉淀。
- JSON 报告应输出 `optimizationPlan`，供后续自动化执行或人工复核。

---

## 回归测试操作
1. 改完 `skill-optimizer` 后，对案例 1-12 跑对应脚本。
2. 实际命中应覆盖期望命中；漏报要修脚本，误报要补例外规则。
3. 案例 1 必须保持高分，防止「逢检必报」的假阳性。
4. JSON 输出必须可被 `json.loads` 解析，并包含维度、红线、Top ROI 修复建议和系统化待确认执行计划。
