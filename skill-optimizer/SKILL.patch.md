## [2026-06-17] v1.5 报告驱动的待确认优化闭环
- **场景**：用户进一步要求使用 skill-optimizer 时，要根据完善后的体检报告，更系统、更详细地给出 Skill 优化计划；并明确纠正：不是直接修改，而是先给体检报告和计划，确认后再修改。
- **本次升级**：
  1. `SKILL.md` 新增「报告驱动优化闭环」，要求优化时按红线、高 ROI、维度深挖、复验沉淀顺序生成待确认计划。
  2. 体检报告模板新增「系统化待确认执行计划」，把问题拆成阶段、优先级、改动对象、具体动作和验收方式。
  3. `health_check.py` 新增 `optimizationPlan` JSON 字段，并在文本报告中输出完整待确认执行计划。
  4. 优化计划覆盖 WARN / FAIL / MANUAL / SKIP，不再只显示 Top 5 修复建议。
  5. `references/golden_set.md` 新增「优化请求缺待确认执行计划」回归案例，防止后续退化。
- **规则**：诊断报告是优化输入；优化交付必须能追溯到报告中的维度、检查项、证据和验收方式。确认前只输出报告和计划；用户确认后才修改、复验和沉淀。
- **关联方法**：#8 IPO 契约 / #11 错误处理 / #12 量化评测 / #14 回归 / #15 Patch 沉淀。

## [2026-06-17] v1.4 完整体检报告与安全稳定专项升级
- **场景**：用户反馈 Skill 体检后不能只输出体检评分，需要完整体检报告，并希望安全性、稳定性等维度更全面。
- **本次升级**：
  1. `SKILL.md` 新增「完整体检报告硬性结构」，明确体检最终回复必须包含结论摘要、全部维度分、红线项、触发审计、安全与稳定性专项、证据明细、ROI 修复、验证和沉淀。
  2. `health_check.py` 从 10 维升级为 11 维，新增「运行稳定性与故障恢复」，并提高「安全与边界」权重。
  3. 安全检查扩展到真实写操作、不可逆动作、敏感信息、最小权限、高风险先问/拒绝。
  4. 稳定性检查扩展到工具失败降级、重复运行/复验、依赖前置条件、MANUAL/SKIP 可见性和失败恢复路径。
  5. 报告模板新增「安全与稳定性专项审计」和「本次改动」章节，避免只报总分。
  6. `references/golden_set.md` 新增「诊断类 Skill 只给总分」和「运行失败无降级路径」两个回归案例。
- **规则**：体检脚本输出是证据来源；最终交付必须展开为完整报告。安全、稳定、未运行项和验证结果不得省略。
- **关联方法**：#3 模板固化 / #11 错误处理 / #12 量化评测 / #14 回归 / #16 可回滚。

## [2026-06-17] v1.3 体检报告详略结构优化
- **场景**：v1.2 已有维度分和逐项证据，但默认文本报告更像检查清单，缺少先结论、再归因、再行动的诊断阅读顺序。
- **本次升级**：
  1. 文本报告新增「诊断结论」，直接说明整体判断、优先关注维度和检查项分布。
  2. 「维度得分」新增每个维度的问题数、关键关注项、证据摘要和红线/失败提示。
  3. 新增「需人工复核 / 未运行」独立区块，把 MANUAL / SKIP 从检查明细里抬出来。
  4. 检查明细补充方法编号，Top ROI 修复建议补充状态、证据和分数影响。
  5. 报告末尾新增复验命令，方便用户直接重跑文本、JSON 和触发冲突审计。
- **规则**：体检报告按「结论摘要 → 红线项 → 维度归因 → 人工复核 → 明细证据 → ROI 动作 → 复验命令」组织；JSON 字段保持兼容，只增加修复项细节。
- **关联方法**：#3 模板固化 / #12 量化评测 / #15 Patch 沉淀 / #17 反馈钩子。

## [2026-06-17] v1.2 维度化体检评分升级
- **场景**：旧版 `health_check.py` 只有单一自动项得分，无法区分结构健康、触发质量、输出稳定和回归能力等不同问题来源。
- **本次升级**：
  1. 将体检模型升级为「总分 + 10 维度分 + 红线项 + Top ROI 修复建议」。
  2. 新增 `--format json`，输出 `totalScore`、`grade`、`blockers`、`dimensions`、`checks`、`topFixes`，便于后续批量审计和自动回归。
  3. 新增 `--skills-root`，把跨 Skill 触发词重叠纳入「触发与路由质量」维度；未传时标记 SKIP 且不扣分。
  4. 同步更新诊断报告模板、自检清单和 Golden Set，覆盖缺 `SKILL.md`、坏 frontmatter、description 过长、触发冲突、无模板、无 Golden Set、真实写操作缺边界等评分案例。
- **规则**：① 总分不替代维度分；② 红线项必须优先修；③ MANUAL/SKIP 不进入自动分母；④ 评分脚本输出必须同时服务人类阅读和 JSON 回归。
- **关联方法**：#5 触发描述 / #6 编排器 / #8 IPO 契约 / #10 脚本化 / #12 Golden Set / #15 Patch 沉淀。
